# GR00T N1.7 × MolmoAct2 SO-101 Fine-Tune Implementation Plan (rev 2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fine-tune `nvidia/GR00T-N1.7-3B` with LoRA on the SO-101 subset of the MolmoAct2-SO100_101 community dataset, entirely on Modal, with wandb train/test-loss + DCGM + FLOPs logging, preemption-safe checkpointing, and publish the resulting checkpoint to HuggingFace.

**Architecture:** A single Modal app (`src/modal_app.py`) builds a pinned Isaac-GR00T image and exposes functions for data prep, smoke test, sweep, training, and publishing. Data lives on Modal volume `gr00t-so101-data`; checkpoints on `gr00t-so101-ckpt`. Training goes through a custom launcher (`src/train_launcher.py`) that reuses `gr00t.experiment.experiment.run` verbatim but injects (a) LoRA adapters, (b) a custom eval-loss callback over a held-out episode split (upstream eval path is dead code — see facts), (c) trainable-only 5-step rolling checkpoints with patched save/load, and (d) a background GPU-metrics sampler thread.

**Tech Stack:** Modal (GPU: H100), Isaac-GR00T @ `ab88b50c718f6528e1df9dcbaf75865d1b604760` (main, 2026-06-30; N1.7), peft LoRA + `category-lora` (installed from git, pin `0a02f3984836e77555a183c140ca51b5aac3376e`) for GR00T's `CategorySpecificLinear` layers, wandb, DCGM 4 / NVML GPM, huggingface_hub.

## Global Constraints

- Base model: `nvidia/GR00T-N1.7-3B` (public, verified accessible with our token).
- Isaac-GR00T pinned to commit `ab88b50c718f6528e1df9dcbaf75865d1b604760`.
- Python 3.10 inside the image (GR00T pin); `torch==2.7.1+cu128`, `flash-attn==2.7.4.post1` prebuilt cp310 wheel (no nvcc needed — comes via `[tool.uv.sources]` + committed `uv.lock`), `torchcodec==0.4.0` (needs FFmpeg).
- `category-lora` installed as `git+https://github.com/jeqcho/category-lora@0a02f3984836e77555a183c140ca51b5aac3376e` (NOT on PyPI — verified 404). API (verified from repo README): `wrap_in_place(model, CategoryLoRAConfig(r, alpha, dropout), target_class_names=["CategorySpecificLinear"]) -> int`, `unload_adapters(model)`.
- Modal secrets already exist: `huggingface-token`, `wandb`. Volumes to create: `gr00t-so101-data`, `gr00t-so101-ckpt`.
- **Checkpoints:** rolling window **every 5 steps, 3 slots** (user requirement) as **trainable-only HF checkpoints** (LoRA/adapter + other trainable params + optimizer + scheduler + RNG + trainer_state; ~hundreds of MB, seconds to write) via patched `Trainer._save` / `_load_from_checkpoint(strict=False)`. Durable "keep" copies of the same trainable-only checkpoints every `keep_steps` (200 sweep / 500 main). Frozen base weights are reconstructed from `base_model_path` at every (re)start, so trainable-only checkpoints are complete for resume.
- **Resume:** decided *inside the container at every (re)start*: `resume = get_last_checkpoint(exp_dir) is not None`. Never pass a static `--resume` flag into a retried function (fresh run with resume=True hard-raises in `Gr00tTrainer.train`; static fresh=True would wipe progress on every retry). `fresh=True` means "wipe exp dir once at operator-invoked launch, then auto-resume mode for retries".
- **LoRA required** (user requirement): peft on DiT `nn.Linear`s; `category-lora` on `CategorySpecificLinear`. Backbone (LLM+visual) frozen. `apply_lora(model) -> model` (mutates in place, returns the model; summary dict logged separately).
- wandb must log: train loss, eval (test) loss, `DCGM_FI_PROF_PIPE_TENSOR_ACTIVE` (1004), `DCGM_FI_PROF_DRAM_ACTIVE` (1005), cumulative FLOPs + FLOP/s (+MFU with a logged caveat: FlopCounterMode misses flash-attn ops; fwd×3 over-counts frozen-backbone backward — treat as monitoring index).
- wandb continuity across preemptions: `WANDB_RUN_ID=<exp_name>`, `WANDB_RESUME=allow` in the train function env.
- Train/test split: **episode-level holdout per source repo** (5%, min 1 episode, seeded RNG) into a sibling `<dataset>_val` root. Eval loss computed by our own `EvalLossCallback` (upstream `val_dataset_path` / `eval_set_split_ratio` / `eval_strategy="steps"` are all dead code — see facts). Never split within an episode.
- `episode_sampling_rate=1.0` explicitly (default 0.1 uses only 10% of timesteps per "epoch"; we want full data and honest epoch math).
- All heavy compute on Modal, never on the Mac. Long-running Modal launches: `modal run --detach` from tmux, logs tee'd to `logs/`.
- Public repo: `.env` never committed (verified gitignored). Commit + push frequently.
- Cost staging: Stage 1 prototype ≈ $50 cap; sweep small; Stage 2 filtered subset ≈ $500 cap. Full 1220-repo mixture out of scope.
- HF publish target: `jeqcho/gr00t-n1.7-so101-molmoact2`.

## Key research facts (re-verified 2026-07-05, rev 2)

- Fine-tune path: `gr00t/experiment/launch_finetune.py` → `FinetuneConfig` → `get_default_config().load_dict(...)` → `experiment.run(config)`.
- **Eval is dead code upstream** (verified in source at the pinned commit): `DatasetFactory.build` (`gr00t/data/dataset/factory.py:40`) asserts `eval_strategy == "no"` and always returns `(train_dataset, None)`; `val_dataset_path` and `eval_set_split_ratio` are declared but consumed nowhere; `Gr00tN1d7.forward` has no `labels`/`return_loss` kwargs so HF `prediction_step` would return `loss=None` even with an eval dataset injected. → Test loss requires our own mechanism (Task 6: `EvalLossCallback` that builds `ShardedSingleStepDataset`s over val roots and calls `trainer.compute_loss` under `no_grad`).
- `Gr00tTrainer.compute_loss` delegates to `Trainer.compute_loss(return_outputs=True)`; model forward returns a dict containing `loss` — so `trainer.compute_loss(model, eval_batch)` is a valid eval-loss path.
- `Gr00tTrainer.train()` raises `ValueError("No valid checkpoint found ...")` if `resume_from_checkpoint=True` with an empty output dir (pinned by upstream test) → auto-detect resume in-container.
- Factory calls `generate_stats(dataset_path)` / `generate_rel_stats(...)` per root at training time with a fingerprint/staleness cache → val roots must carry stats files that (a) match the train root byte-for-byte and (b) pass the freshness check, else eval normalization silently diverges. Smoke test must assert stats identity after one full launch.
- SO-101: `--embodiment-tag NEW_EMBODIMENT` + `examples/SO100/so100_config.py`; GR00T-flavored LeRobot v2 (v2 layout + `meta/modality.json`, video keys exactly `front`/`wrist` with per-repo `original_key` remapping, state/action `single_arm` [0:5] + `gripper` [5:6], `annotation.human.task_description` ← `task_index`).
- LeRobot v3 → v2: `scripts/lerobot_conversion/convert_v3_to_v2.py` (argparse: `--repo-id`, `--root`, `--force-conversion`; run via `uv run --project scripts/lerobot_conversion`). It downloads from the Hub itself and nests output at `<root>/<user>/<repo>`; prepare code must handle the nesting.
- N1.7 has zero LoRA code; `peft==0.17.1` already in deps. `CategorySpecificLinear` at `gr00t/model/modules/embodiment_conditioned_mlp.py:59`; DiT at `gr00t/model/modules/dit.py`. `MODEL_REGISTRY` is a plain dict keyed by config class; `run()` calls `pipeline.return_model()` / `return_dataset()` / `return_collator()` — class-level monkeypatching is mechanically sound.
- Manifest `allenai/MolmoAct2-SO100_101-Dataset`: `repo_list.json` = 1220 `"lerobot:<user>/<repo>"` strings, no metadata; `language_annotations/<user>/<repo>/tasks_annotated.parquet` = per-`episode_index` `task` strings (37,459 rows). Filtering requires per-repo `meta/info.json` fetch.
- `whosricky/so101-megamix-v1` (Stage 1): NOT in manifest; LeRobot v3.0, `so101_follower`, 400 eps / 179,166 frames / 30 fps / 8 tasks; cameras `front`, `top`, `gripper`(wrist) @ 640×480; state/action float32[6].
- DCGM: fields 1004/1005; `datacenter-gpu-manager-4-cuda12` exists in NVIDIA's ubuntu2204 apt repo (verified); bindings at `/usr/share/datacenter-gpu-manager-4/bindings/python3`; needs `nv-hostengine` (root — fine on Modal); **unverified under gVisor** → smoke-test `dcgmi dmon -e 1004,1005 -c 3`. Fallback tier 2: NVML GPM on H100 (`NVML_GPM_METRIC_ANY_TENSOR_UTIL=5`, `NVML_GPM_METRIC_DRAM_BW_UTIL=10`) logged under the DCGM key names (legitimate equivalents). Tier 3: `nvmlDeviceGetUtilizationRates` logged under **different** keys (`gpu/util`, `gpu/mem_util`) — semantically different, never masquerade.
- FLOPs: HF `state.total_flos` = 0 for GR00T (no `input_ids`). Calibrate once with `FlopCounterMode` (fwd×3), log `flops_total`, `flops_per_sec`, `mfu_approx`.
- Modal: root under gVisor; volumes need `.commit()`; preemption → `modal.Retries(max_retries=24, initial_delay=60, backoff_coefficient=1.0)` + in-container auto-resume makes timeout/preemption just another retry.

### Task 4: Checkpoint patches + callbacks (`src/ckpt_patches.py`, `src/callbacks.py`)

**`ckpt_patches.py` produces** (design, not fallback — full 6 GB saves every 5 steps would dominate wall-clock):
- `patch_trainable_only_checkpointing(TrainerCls)`:
  - `_save(output_dir, state_dict=None)` override: write ONLY `{k: v for k,v in state_dict if k in trainable_keys}` as `trainable.safetensors` + `trainable_keys.json`; skip full model weights. HF still writes optimizer.pt (trainable-only states — small), scheduler, RNG, trainer_state.json around it.
  - `_load_from_checkpoint(resume_path)` override: base weights already loaded from `base_model_path` by pipeline setup + LoRA modules already attached (init A kaiming/B zero) → `load_state_dict(load_file(trainable.safetensors), strict=False)`; assert loaded key set == saved key set; then let HF load optimizer/scheduler/RNG/state as usual.
  - Compatibility guard: assert not deepspeed, single GPU, `save_only_model=False`.
- `wipe_or_resume(exp_dir, fresh: bool) -> bool`: if fresh → rmtree once (operator launch only, guarded by a `.fresh_done` marker so retries don't re-wipe); return `get_last_checkpoint(exp_dir) is not None`.

**`callbacks.py` produces:**
- `FlopsCallback(flops_per_step)` — `on_log`: `flops_total`, `flops_per_sec`, `mfu_approx` (H100 BF16 ≈ 989e12·n_gpus) merged into `logs`. `calibrate_flops(trainer)` measures one batch with `FlopCounterMode` (fwd×3), caveat logged into wandb config.
- `EvalLossCallback(trainer, val_roots, every_steps, max_batches=32)` — builds `ShardedSingleStepDataset` per val root (same modality configs/embodiment tag as train, `episode_sampling_rate=1.0`) + pipeline collator + DataLoader(eval_batch_size); `on_step_end` when `step % every_steps == 0`: `model.eval()`, `torch.no_grad()`, mean of `trainer.compute_loss(model, batch)` over ≤max_batches, `wandb.log({"eval/loss": v}, commit=False)`, back to `model.train()`. Also runs at step 1 (baseline) and on_train_end.
- `KeepCheckpointCallback(keep_steps, keep_dir)` — `on_save`: if `step % keep_steps == 0`: copytree rolling checkpoint → `keep_dir/checkpoint-<step>` (trainable-only ⇒ cheap).
- `VolumeCommitCallback(volume)` — `on_save`: `volume.commit()`; no-op outside Modal.

- [ ] Write both modules.
- [ ] Local unit-check (uv script, CPU, tiny model with some frozen params + HF Trainer on random data): rolling dir ≤3 checkpoints; checkpoint contains trainable-only weights; kill after step 12 → relaunch resumes from checkpoint-10 with matching optimizer state; keep-dir gets a copy at keep_steps; flops keys present in logs.

Mechanism:
```python
import gr00t.experiment.experiment as exp
from gr00t.model import MODEL_REGISTRY
from gr00t.configs.model.gr00t_n1d7 import Gr00tN1d7Config

class PatchedTrainer(exp.Gr00tTrainer):
    def __init__(self, *a, **kw):
        super().__init__(*a, **kw)
        self.add_callback(FlopsCallback(calibrate_flops(self)))
        self.add_callback(EvalLossCallback(self, val_roots, eval_steps, eval_batches))
        self.add_callback(KeepCheckpointCallback(keep_steps, keep_dir))
        self.add_callback(VolumeCommitCallback(CKPT_VOL))
patch_trainable_only_checkpointing(PatchedTrainer)
exp.Gr00tTrainer = PatchedTrainer

PipelineCls = MODEL_REGISTRY[Gr00tN1d7Config]
_orig = PipelineCls.return_model
PipelineCls.return_model = lambda self: apply_lora(_orig(self), r=args.lora_r)
```

## Risks & mitigations

1. **DCGM under gVisor unverified** → 3-tier fallback; tier 2 GPM keeps DCGM key names (equivalent metrics); tier 3 uses honest distinct keys.
2. **Trainable-only checkpoint/resume is custom code** → it IS the design (not a fallback); proven by local unit test (Task 4) AND live kill-and-resume drill (Task 7) before any long run.
3. **Eval correctness (normalization)** → stats generated once on train root, copied to val root, byte-compare asserted after smoke run.
4. **Community repos malformed** → loadability filter at prep; failures dropped+logged; never crash sweep/main.
5. **Preemption** → in-container auto-resume + retries=24 + WANDB_RUN_ID continuity; drill-proven.
6. **flash-attn / env breakage** → committed `uv.lock` + pinned commit; env_check imports everything on H100 before GPU spend.
7. **FLOPs/MFU accuracy** → logged as `mfu_approx` with caveat in wandb config.
8. **Renumbering bugs in split** → loader validation on both roots at prep time + eval baseline at step 1 sanity-checks magnitudes.
